webpack

js工具
js工具
发布日期:   2020-09-07
webpack 是一个模块打包工具,基于nodejs,只支持js和JSON模块

一、基础的一些配置(entry,mode,output等)

1.entry 入口文件的设置

  • 使用字符串 一个入口,一个出口
entry:'./src/index.js'
  • 使用数组 可以是多个入口，但是一个出口
entry:['./src/index.js','./src/other.js']
  • 使用对象 可以是多个入口，多个出口口 (多页面应用)
  // 3个属性名,对应了3个出口文件
entry:{ 
  main:'./src/index.js',
  other:'./src/other.js',
  others:['./src/test.js','./src/test1.js']
}

2. mode 模块打包的模式

默认值是 production 生产环境

3. output 出口文件

output:{
  //生成的出口问件 目录,绝对路径
  path:path.resolve(__dirname,'./dist'),
 //  构建的文件
  // 1.一个 出口文件 使用的是字符串的格式
  filname:'main.js'
  // 2. 出口文件 使用的是占位符的格式
  filname:'[name].js'
  // 3. 出口文件 使用的是hash(长度为6)+占位符的格式
  // 修改 一个文件,打包后所有文件的hash都改变了,整个项目的缓存文件,都将失效
  // 多个入口文件，hash值都是一样的
  filname:'[name]_[hash:6].js'
  // 4. 出口文件 使用的是chunkhash(长度为6)+占位符的格式  
  //  chunkhash 对于代码块 互不影响且不存在引用关系的代码块,改变一个文件,当前文件的hash值改变
  //另一个文件的hash值不变, 且不会再次打包
  filname:'[name]_[chunkhash:6].js'
 // 5. 出口文件 使用contenthash(长度为6)+占位符的格式  
  //  contenthash 对于一个文件，在同时引用js文件和css文件时，当js文件发生改变时，css文件不变,
  // 则 js文件的hash值改变，而css文件不改变
  filname:'[name]_[contenthash:6].js'
 //  指定静态资源路径CDN,需要把文件上传到服务器
  publicPath:''   
}

4. context 基础目录

//上下文   项目打包的相对路径 必须是绝对路径
 // process.cwd() 是当前Node.js进程执行时的文件夹目录——工作目录，保证了文件在不同
 //的目录下执行时，路径始终不变
 context: process.cwd()

5. devtool 中的sourceMap

sourceMap 打包后的文件和原文件之间的关系,如果开启,可以定位到原文件,不安全
  • mode==development 默认开启
  • mode==production 一般不建议开启,若要开启使用 cheap-module-eval-source-map
devtool:"cheap-module-source-map"

五、 babel 处理ES6+

5.1 babel-loader 的使用

1.下载 babel-loader
npm i -D babel-loader @babel/core  @babel/preset-env 
npm i -S  @babel/polyfill
2.使用
  /*
  * babel-loader  只是一个桥梁,建立webpack与babel之间的桥梁,不会进行转换
  * exclude 排除这些文件不需要
  * @babel/core 语法转换的核心API
  * @babel/preset-env 语法转换规则 可以将es6+转化为es5,但是对于Promise等不识别的浏览器,也会报错
  * polyfill 垫片 es6+的库, 将es6+的语法库 直接导入到使用到es6+语法的文件,使其支持es6+的写法
  * polyfill 会污染全局对象
  * 1. polyfill 简单粗暴使用,在所有使用到es6模块的文件,文件会变得很大,不适用
  * import '@babel/polyfill'
  * 2. 按需引入polyfill
  *  useBuiltIns: "usage"
  * targets 目标浏览器
  */
module:{
  rules:[
    {
      test:/\.js$/,
      exclude:/node_modules/,
      use:{
        loader:'babel-loader',
        options:[
          [
            '@babel/preset-env',
            {
              targets: {
                edge: "17",
                firefox: "60",
                chrome: "67",
                safari: "11.1"
              },
              corejs: 2,//新版本需要指定核⼼库版本
              useBuiltIns: "usage"
            }
          ]
        ]
      }
    }
  ]
}

六、性能优化

6.1 优化开发体验(优化构建速度、优化使用体验、提升效率)

  • include 缩小loader的查找方位 (exclude 排除哪里不查找)
module:{
  rules:[
    {
      test:/\.css$/,
      use:['style-loader','css-loader'],
      include:path.resolve(__dirname,'.src')
    }
  ]
}
  • resolve 模块解析
/**
 *  resolve.modules 直接设置成当前项目的 node_modules 路径,减小查找范围
 *  resolve.alias 设置别名,减少查找过程,
 *      react 直接指向了react.production.min.js,减少了查找时间
 *  resolve.extensions 自动添加文件的后缀,尽量不要使用
*/
module.exports={
  resolve:{
    modules: [path.resolve(__dirname, "./node_modules")],
    alias:{
      react:path.resolve(__dirname,"./node_modules/react/umd/react.production.min.js"),
      "react-dom": path.resolve(__dirname,"./node_modules/react-dom/umd/react-dom.production.min.js")
    },
    extensions:['.js','.ts'],
  }
}
  • 使用Dllplugin 优化构建性能 需要手动引入
    • 项⽬中引⼊了很多第三⽅库，这些库在很⻓的⼀段时间内，基本不会更新，打包的时候分开打包来提升打包速度，⽽DllPlugin动态链接库插件，其原理就是把⽹⻚依赖的基础模块抽离出来打包到dll⽂件中，当需要导⼊的模块存在于某个dll中时，这个模块不再被打包，⽽是去dll中获取。
const { DllPlugin } = require("webpack");
module.exports = {
  output: {
    path: path.resolve(__dirname, "./dll"),
    filename: "[name].dll.js",
    library: "react" //和下方的 name名字一样
  },
  plugins: [
    new DllPlugin({
      // manifest.json⽂件的输出位置
      path: path.join(__dirname, "./dll", "[name]-manifest.json"),
      // 定义打包的公共vendor⽂件对外暴露的函数名
      name: "react"
    })
  ]
}

在package.json中添加,构建动态链接库
"dev:dll": "webpack --config ./build/webpack.dll.config.js"

七、小问题总结

7.1 webpack中chunk和bundle中的关系

  • chunk 是webpack对源文件进行打包时,根据引用关系生成 chunk文件,并对其进行一些操作,不可以在浏览器运行
  • 处理好的chunk文件,最后输出成 bundle,可以直接再浏览器中运行

7.2 file-loader和url-loader的区别

  • file-loader 把打包⼊⼝中识别出的资源模块，移动到输出⽬录，并且返回⼀个地址名称
  • url-loader 包含了file-loader,但是对jpg格式的模块,会将其进行压缩,并打包到js中,大体积不合适
